--- Slide 4: #1. Avoid TRUNC(date) in WHERE ---
Goal: Speed up date filters by using range predicates instead of TRUNC(),
which disables index/partition usage on the date column.
[SNIPPET]
Outcome: 80 s -> 0.05 s (1600x). Range condition enabled index on EFF_DT
and partition pruning. SAS showed identical benefit when filter passed through
as date range. Biggest single improvement in the benchmark.
Takeaway: Never wrap indexed date columns in functions inside WHERE clauses.

--- Slide 5: #2. Avoid TRUNC(date) in GROUP BY ---
Goal: Group by the date column directly instead of TRUNC(date_col) to allow
index usage and avoid per-row function overhead.
[SNIPPET]
Outcome: 7.9 s -> 2.3 s (3.5x). Direct grouping utilized index on DW_BUS_DT.
Modest but consistent improvement. SAS showed similar pattern.
Takeaway: If data is already at the needed granularity, group by the column directly.

--- Slide 6: #3. Skip Unnecessary CTE ---
Goal: Eliminate overhead of trivial CTEs. For simple one-off aggregations,
a direct query avoids extra parsing and execution steps.
[SNIPPET]
Outcome: 11 s -> 1.3 s (8x). The CTE wrapper added overhead without benefit
for a single-use aggregation. SAS performance was analogous.
Takeaway: Use direct queries for simple aggregations; reserve CTEs for reuse.

--- Slide 7: #4. Parallel Query (PARALLEL hint) ---
Goal: Accelerate large aggregations via Oracle parallel execution with
PARALLEL(table, 4) hint.
[SNIPPET]
Outcome: No gain (2.1 s serial vs 3.9 s parallel). SESSIONS_PER_USER=3 limited
available threads; coordination overhead outweighed any benefit.
Takeaway: Parallel hints require sufficient session slots. Check DBA limits first.

--- Slide 8: #5. Client Fetch Tuning (ARRAYSIZE) ---
Goal: Reduce network round-trips for large result sets by increasing the
Python cursor arraysize from 100 (default) to 5000.
[SNIPPET]
Outcome: 20.8 s -> 1.3 s (16x) for 200k rows. Fewer round-trips between client
and DB. SAS already uses large fetch blocks (~1.3-1.5 s baseline).
Takeaway: Always set arraysize >= 5000 for bulk data retrieval in Python.

--- Slide 9: #6. Bind Variables ---
Goal: Reduce SQL parsing overhead for repetitive lookups by using bind
parameters instead of literal values in looped queries.
[SNIPPET]
Outcome: ~10x estimated improvement for 500 single-row lookups. Bind parameters
allow Oracle to reuse execution plans. SAS uses binds automatically.
Takeaway: For batch/looped queries, always use bind parameters.

--- Slide 10: #7. Partition Pruning (TO_CHAR vs date range) ---
Goal: Enable Oracle partition pruning by filtering with native date range
instead of TO_CHAR() on the partition key.
[SNIPPET]
Outcome: 54 s -> 0.3 s (180x). TO_CHAR forced full-partition scan; date range
restricted scan to January 2024 partition only. Second-highest gain.
Takeaway: Never apply functions to partition key columns in WHERE clauses.

--- Slide 11: #8. RESULT_CACHE Hint ---
Goal: Leverage Oracle's server-side result cache for repeated identical
queries using the /*+ RESULT_CACHE */ hint.
[SNIPPET]
Outcome: 10.5 s -> 0.47 s avg across 5 runs (22x). First run is normal speed;
subsequent runs return cached results sub-second. Cache invalidated on DML.
Takeaway: Use RESULT_CACHE for read-heavy, repeatedly executed queries.

--- Slide 12: #9. Batch Fetching (fetchone vs fetchmany) ---
Goal: Minimize application-level overhead by fetching rows in batches of 5000
instead of one-by-one with fetchone().
[SNIPPET]
Outcome: ~15x estimated improvement for 100k rows. Reduces Python-to-DB call
count dramatically. SAS handles this internally with bulk fetch.
Takeaway: Use fetchmany() or set large arraysize; never loop with fetchone().
