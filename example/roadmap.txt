================================================================================
  dtrack Web UI — Roadmap
================================================================================


WHAT THE USER SEES
------------------

A single-page app at localhost:8080. Left sidebar shows the pair list.
Main area walks through the workflow as a stepped pipeline. Each step has
a panel that shows status, output, and controls.

================================================================================
  STEP-BY-STEP: WHAT EACH PANEL DOES
================================================================================


STEP 0 — Database
─────────────────

  State:  ✓ ready  /  ○ not initialized

  What it shows:
    Database path, table count, pair count, last modified time.

  Controls:
    [ Initialize ]     Creates project.db (or reinitializes with --force)
    [ Refresh ]        Runs schema migration on existing db

  On click:
    POST /api/init
    Returns: { status, tables, pairs }


STEP 1 — Columns
─────────────────

  State:  ✓ loaded  /  ○ pending

  What it shows:
    Table listing each source table and its column count.
    Expandable per-table view showing column names and data types.

  Controls:
    [ Upload CSV ]     File picker for colmap CSV(s)
    [ Discover ]       Runs live discovery (Oracle/Athena)

  On click:
    POST /api/columns/load    { csv: <uploaded file> }
    POST /api/columns/discover
    Returns: { tables: [{ name, columns: [{ name, type }] }] }

  Live output:
    WebSocket streams discovery progress line by line.


STEP 2 — Extract Row Counts
────────────────────────────

  State:  ✓ done  /  ◐ running  /  ○ pending

  What it shows:
    Two sub-panels side by side:

    ┌─ SAS (Oracle/PCDS) ────────┐  ┌─ Athena (AWS) ─────────────┐
    │  Status: generated         │  │  Status: running 3/4       │
    │  File: ./sas/extract.sas   │  │  aws.cust_daily ✓  3.2s   │
    │                            │  │  aws.txn_monthly ✓  2.1s  │
    │  [ Generate SAS ]          │  │  aws.acct_balance ◐  ...   │
    │  [ Download .sas ]         │  │  aws.email_events ○        │
    │                            │  │                            │
    │                            │  │  [ Run Extraction ]        │
    └────────────────────────────┘  └────────────────────────────┘

  Controls:
    [ Generate SAS ]    POST /api/extract/sas   { type: "row" }
                        Returns SAS file path. Offers download link.
                        (User runs it externally in SAS EG)

    [ Run Extraction ]  POST /api/extract/aws   { type: "row" }
                        Runs live. WebSocket streams per-table progress.

  After SAS runs externally, user clicks [ I've run the SAS ] to mark done.

    │  ☐ 2023-08-14  -2            ☐ 2023-04-15  aws only             │

STEP 5 — Match Columns                                   ** INTERACTIVE **
───────────────────────

  State:  ✓ done  /  ◐ mapping  /  ○ pending

  What it shows:
    Per pair, a three-column layout:

    ┌─ cust_daily ─────────────────────────────────────────────────────┐
    │                                                                  │
    │  Auto-matched: 8 columns                                        │
    │                                                                  │
    │  LEFT (pcds)              MAPPING           RIGHT (aws)          │
    │  ────────────             ───────           ──────────           │
    │  CUSTOMER_ID NUMBER    ←→ auto           customer_id bigint     │
    │  REGION      VARCHAR   ←→ auto           region      string     │
    │  RPT_DT      TIMESTAMP ←→ auto           rpt_dt      date       │
    │  STATUS      CHAR(1)   ←→ auto           status      string     │
    │  AMT         NUMBER    ←→ [________]     amount      double     │
    │  CUST_STATUS VARCHAR   ←→ [________]     cust_stat   string     │
    │                                           load_dt     timestamp  │
    │                                                                  │
    │  Drag right column names into the input boxes, or type them.    │
    │                                                                  │
    │  [ Save Mappings ]                                               │
    └──────────────────────────────────────────────────────────────────┘

  Unmapped left columns have an empty input box. Unmapped right columns

  Controls:
    GET  /api/columns/{pair}      Returns auto-match results + unmatched
    POST /api/columns/match       Saves final col_map to config + db

================================================================================
  API ROUTES
================================================================================

  Database & Config
    POST   /api/init                    Initialize database
    GET    /api/config                  Return current config JSON
    PUT    /api/config                  Update config (partial merge)
    POST   /api/query                   Run SQL against project.db

  Columns
    POST   /api/columns/load            Upload and load column CSV
    POST   /api/columns/discover        Discover columns live
    GET    /api/columns/{pair}           Get auto-match results for pair
    POST   /api/columns/match           Save column mappings

  Extract
    POST   /api/extract/sas             Generate SAS file
    POST   /api/extract/aws             Run Athena extraction (async)

  Load
    POST   /api/load/row                Load row CSVs from folder
    POST   /api/load/col                Load col CSVs from folder

  Compare
    POST   /api/compare/row             Run row comparison (returns results)
    POST   /api/compare/col             Run col comparison (returns results)

  Reports
    GET    /api/report/row              Serve row comparison HTML
    GET    /api/report/col              Serve col comparison HTML

  Pairs
    GET    /api/pairs                   List all pairs with status
    GET    /api/pairs/{name}            Get pair details + comparison state

  WebSocket
    WS     /ws/progress                 Live log streaming during long ops

================================================================================
  TECH STACK
================================================================================

  Backend:
    fastapi           ASGI framework, async-ready
    uvicorn           ASGI server
    jinja2            HTML templating (index.html shell only)

  Frontend:
    Vanilla JS        No build step, no npm, no node_modules
    Fetch API         For all /api/ calls
    WebSocket API     For live progress streaming
    CSS Grid          Layout

  No React, no Vue, no webpack. The frontend is ~500 lines of JS that
  fetches JSON and renders HTML strings. The existing html_export.py
  already produces self-contained HTML — we just embed it.
